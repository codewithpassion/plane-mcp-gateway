## Project Overview

This is a **Model Context Protocol (MCP) server** that provides remote OAuth authentication using Clerk. It's built on Cloudflare Workers and uses the `@cloudflare/workers-oauth-provider` library to implement OAuth 2.1 compliant authentication. The server can be deployed to Cloudflare Workers and exposes MCP tools via HTTP+SSE or Streamable-HTTP protocols.

### Key Responsibilities
- Acts as an OAuth 2.1 server for MCP clients
- Acts as an OAuth 2.1 client to Clerk's OAuth service
- Manages MCP tool definitions and conditional access based on authenticated user roles
- Handles authentication state, CSRF protection, and session binding for security

## Architecture

### Three-Layer Architecture

**1. OAuth Layer** (`src/clerk-handler.ts`, `src/workers-oauth-utils.ts`)
- Implements OAuth 2.1 authorization flow using `@cloudflare/workers-oauth-provider`
- Manages endpoints: `/authorize`, `/callback`, `/register`, `/token`
- Clerk acts as the upstream OAuth provider; this server acts as the OAuth server for MCP clients
- Token contains encrypted user context (userId, sessionId, email, firstName, lastName, role, metadata)

**2. MCP Server Layer** (`src/index.ts`)
- Extends `McpAgent` from the `agents` library (Durable MCP)
- Defines tools using the MCP SDK with Zod validation
- Tools receive user context via `this.props` (authenticated user info)
- Supports conditional tool availability based on user role (example: `generateImage` tool restricted to `ALLOWED_ROLES`)
- Exposes two protocol endpoints:
  - `/sse` - deprecated Server-Sent Events protocol
  - `/mcp` - current Streamable-HTTP protocol (recommended)

**3. Utility Layer** (`src/utils.ts`)
- `getUpstreamAuthorizeUrl()` - constructs OAuth authorization URLs
- `fetchUpstreamAuthToken()` - exchanges auth code for access token
- `Props` type - defines authenticated user context (userId, sessionId, email, firstName, lastName, role, metadata)

**4. Plane Tool Suite** (`src/plane/`)
- ~109 MCP tools wrapping the Plane REST API (`https://api.plane.so/api/v1/...`), transliterated from the upstream Python `plane-mcp-server` and `plane-sdk` (no official OpenAPI spec exists).
- Layout:
  - `client.ts` - `planeFetch()` with retry/backoff, auth via `X-Api-Key` or `Authorization: Bearer`, trailing-slash URL convention matching the Python SDK
  - `context.ts` - `getPlaneContext(env)` returns `{ config, workspaceSlug }`, cached per-isolate
  - `errors.ts` - `PlaneError` / `HttpError` / `ConfigurationError`
  - `env.d.ts` - augments `Cloudflare.Env` with `PLANE_*` keys (NOT regenerated by `wrangler types`)
  - `types/` - hand-ported TS interfaces for request/response models (one file per resource)
  - `resources/` - one module per Plane API resource (projects, work_items, cycles, modules, pages, epics, labels, states, milestones, initiatives, intake, work_logs, work_item_types, work_item_properties, users, workspaces). The `work_items` resource consolidates activities/comments/links/relations sub-endpoints to mirror the Python SDK.
  - `tools/` - one MCP tool module per Python file; `tools/index.ts` exports `registerPlaneTools(server, ctx)` which calls every per-module `register*Tools` registrar
  - `tools/_helpers.ts` - `toolResult(fn)` wrapper (uniform JSON-text return + error handling) and `stripNullish(obj)` (mirrors pydantic `model_dump(exclude_none=True)`)
- Tool names, descriptions, and parameter names match the Python `plane-mcp-server` verbatim so prompts written against the upstream server keep working.
- Step 1 auth model: a server-wide Plane API key from env vars (NOT per-user). All Clerk-authenticated MCP clients see the same Plane workspace. Per-user OAuth-to-Plane mapping and `X-Api-Key` / `X-Workspace-slug` header passthrough are deferred.
- If `PLANE_API_KEY` or `PLANE_WORKSPACE_SLUG` is missing at startup, `MyMCP.init()` logs a warning and skips Plane registration; the rest of the server still works.

### Data Flow
1. Client connects to `/mcp` or `/sse`
2. OAuth check redirects to `/authorize`
3. User approves access → CSRF & session binding cookies created
4. Redirect to Clerk → Clerk redirects to `/callback`
5. Server exchanges code for Clerk tokens (access_token + id_token)
6. Server verifies JWT and extracts user data from claims
7. Server creates MCP token with user props
8. Client receives token and can invoke tools
9. Tools execute with user context available via `this.props`

## Security Implementation

### OAuth State Management (`src/workers-oauth-utils.ts`)
- **KV Storage**: OAuth state tokens stored in Cloudflare KV with 600s TTL
- **Session Binding**: State token hashed and bound to browser session via `__Host-CONSENTED_STATE` cookie
- **CSRF Protection**: Per-form CSRF tokens stored in `__Host-CSRF_TOKEN` cookie with 10-minute TTL, validated on form submission
- **One-time Use**: State and CSRF tokens deleted/invalidated after use (RFC 9700 compliance)
- **URL Validation**: Input URLs validated for XSS attacks using allowlist (http/https only) + HTML escaping

### Access Control
- Basic tools (`add`, `userInfo`) available to all authenticated users
- Plane tools (~109) available to all Clerk-authenticated users (no role gating in step 1)
- Restricted tools (e.g., `generateImage`) controlled via role-based access control
- User role from Clerk available in `this.props.role` for conditional logic
- Roles configured in `ALLOWED_ROLES` set in `src/index.ts`
- User roles set in Clerk Dashboard via `public_metadata.role` field

### Secrets Management
Wrangler secrets (do not commit):
- `CLERK_CLIENT_ID` - Clerk OAuth application client ID
- `CLERK_CLIENT_SECRET` - Clerk OAuth application client secret
- `CLERK_SECRET_KEY` - Clerk secret key for JWT verification (starts with `sk_`)
- `CLERK_FRONTEND_API` - Clerk frontend API URL (e.g., `https://your-subdomain.clerk.accounts.dev`)
- `COOKIE_ENCRYPTION_KEY` - Random string for signing cookies (generate: `openssl rand -hex 32`)
- `PLANE_API_KEY` - Plane workspace API token (Settings → API Tokens in Plane)
- `PLANE_WORKSPACE_SLUG` - Plane workspace slug (the path segment after `https://app.plane.so/`)

Non-secret env vars (declared in `wrangler.jsonc` `vars`):
- `PLANE_BASE_URL` - Plane API base URL (default `https://api.plane.so`; override for self-hosted)
- `PLANE_INTERNAL_BASE_URL` - optional, preferred over `PLANE_BASE_URL` for server-to-server calls

## Development

### Setup
1. Install dependencies: `bun install`
2. Create Clerk application at https://dashboard.clerk.com
3. Create OAuth application in Clerk Dashboard (redirect URI: `http://localhost:8788/callback`)
4. Copy `.env.example` to `.env` and fill in Clerk + Plane credentials
5. Generate `COOKIE_ENCRYPTION_KEY`: `openssl rand -hex 32`
6. Get a Plane API key: Plane workspace → Settings → API Tokens → create token, set `PLANE_WORKSPACE_SLUG`
7. Set user roles in Clerk Dashboard: Users → [select user] → Metadata → Public → `{"role": "admin"}`

### Common Commands
- `bun run dev` (or `bun start`) - Run locally on `http://localhost:8788`
- `bun run type-check` - Check TypeScript types without emitting
- `bun run cf-typegen` - Generate Cloudflare Worker types (only refreshes vars from `wrangler.jsonc`; Plane `Env` augmentation lives in `src/plane/env.d.ts` and is not regenerated)
- `bun run lint` - Biome check
- `bun run deploy` - Deploy to Cloudflare Workers

### Testing
Test locally with MCP Inspector:
```bash
bun run dev
# In another terminal:
bunx @modelcontextprotocol/inspector@latest
# Enter: http://localhost:8788/sse (deprecated) or http://localhost:8788/mcp
```

Then use the Inspector to authenticate via Clerk and test tools.

### Adding New Tools
1. **Top-level tools**: in `src/index.ts`, add inside `MyMCP.init()` using `this.server.tool()`. Use `this.props` for Clerk user context. Use Zod for input validation. Optionally gate by role: `if (this.props!.role && ALLOWED_ROLES.has(this.props!.role)) { ... }`.
2. **Plane tools**: add to the matching file under `src/plane/tools/<resource>.ts`. Each tool follows the shape
   ```ts
   server.tool("snake_case_name", "description", { /* zod shape */ }, async (params) =>
     toolResult(() => resource.method(ctx.config, ctx.workspaceSlug, params)),
   );
   ```
   Add the underlying HTTP call to `src/plane/resources/<resource>.ts` using `planeFetch(config, method, path, { params?, body? })`. URL paths must NOT include `/api/v1/` (client adds it) and don't need a trailing slash (client adds one). Use `stripNullish(body)` on POST/PATCH bodies to mirror Python's `model_dump(exclude_none=True)`.
3. **New resource**: also wire its `register*Tools` into `src/plane/tools/index.ts`.

### Production Deployment
1. Create Clerk OAuth application with production redirect URI: `https://mcp-clerk-oauth.<your-subdomain>.workers.dev/callback`
2. Set secrets:
   - Clerk: `wrangler secret put CLERK_CLIENT_ID`, `CLERK_CLIENT_SECRET`, `CLERK_SECRET_KEY`, `CLERK_FRONTEND_API`, `COOKIE_ENCRYPTION_KEY`
   - Plane: `wrangler secret put PLANE_API_KEY`, `wrangler secret put PLANE_WORKSPACE_SLUG`
3. Create KV namespace: `wrangler kv namespace create "OAUTH_KV"` and update `wrangler.jsonc` with ID
4. Deploy: `bun run deploy`
5. Review [Securing MCP Servers](https://github.com/cloudflare/agents/blob/main/docs/securing-mcp-servers.md)

## Configuration

### wrangler.jsonc
- `compatibility_date: "2025-03-10"` - Cloudflare runtime version
- `compatibility_flags: ["nodejs_compat"]` - Enable Node.js compatibility
- `MCP_OBJECT` Durable Object binding - stores MCP server state
- `OAUTH_KV` KV namespace - stores OAuth state tokens
- `AI` binding - access to Cloudflare AI models (for generateImage tool)
- Dev port: 8788

### Environment Types
- `worker-configuration.d.ts` is auto-generated by `bun run cf-typegen` from `wrangler.jsonc`. **Do not regenerate it without preserving secrets**: secrets stored only in `.env`/`wrangler secret` are NOT seen by `wrangler types` and will be dropped from `Cloudflare.Env` on regeneration. The current file was generated when Clerk vars were declared in wrangler config; re-running `cf-typegen` today removes them.
- Plane env vars are NOT in `worker-configuration.d.ts`. They live in `src/plane/env.d.ts` which type-augments `Cloudflare.Env` and survives `cf-typegen` regenerations.

## Important Security Notes

This is a **demo template**. Before production:
- Implement rate limiting (auth attempts, tool invocations)
- Add logging and monitoring
- Review Cloudflare's [Securing MCP Servers guide](https://github.com/cloudflare/agents/blob/main/docs/securing-mcp-servers.md)
- Validate all user inputs in tools
- Consider token rotation strategies for long-lived MCP sessions
- Test OAuth attack vectors (CSRF, code reuse, state mismatches)

## Common Gotchas

1. **SSE vs Streamable-HTTP**: SSE (`/sse`) is deprecated; use `/mcp` for new clients
2. **KV Namespace ID**: Must be set in `wrangler.jsonc` before deployment (placeholder: `<Add-KV-ID>`)
3. **User Roles**: Set via Clerk Dashboard → Users → Metadata → Public → `{"role": "admin"}`
4. **ALLOWED_ROLES**: Default roles are `"admin"` and `"premium"` - customize in `src/index.ts`
5. **Clerk Frontend API**: Must include full URL with protocol (e.g., `https://your-subdomain.clerk.accounts.dev`)
6. **Durable Object Migrations**: New class names require migration tags in `wrangler.jsonc`
7. **Plane URL paths**: pass paths to `planeFetch` WITHOUT the `/api/v1/` prefix and WITHOUT a trailing slash — the client adds both. Mismatches usually surface as 404s with HTML bodies.
8. **Plane work items endpoint**: the SDK uses `/work-items/` (hyphenated), NOT `/issues/`. Some older Plane docs still say `issues` — trust the Python SDK paths.
9. **`wrangler types` drops secrets**: see the Environment Types section above. Augment `Cloudflare.Env` via `src/plane/env.d.ts`-style `.d.ts` files rather than regenerating.
10. **No Plane env vars set**: `MyMCP.init()` catches `PlaneError` and skips registration with a `console.warn` — the server still serves `add`, `userInfo`, and (for permitted roles) `generateImage`.
